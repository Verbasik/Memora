Проведи рефлексию над накопленными сессиями: найди повторяющиеся паттерны, синтезируй инсайты и промотируй только подтверждённое знание.

**Ключевое отличие от memory-consolidate:**
`consolidate` переносит факты из сессий 1-к-1. `reflect` синтезирует знания ПОПЕРЁК нескольких сессий: N сессий → 1 инсайт. Не запускай reflect вместо consolidate — они дополняют друг друга.

## Шаг 1. Отбор сессий для рефлексии

Прочитай все файлы в `memory-bank/.local/SESSIONS/`.

Отфильтруй:
- Уже помеченные строкой `reflected:` в front matter или в теле → **пропустить**.
- Файлы моложе 1 дня → **пропустить** (сессия ещё "горячая").

Если необработанных сессий < 2 — **завершить с сообщением**: «Недостаточно сессий для рефлексии (нужно ≥ 2). Запусти позже.»

## Шаг 2. Тематическая кластеризация

Для каждой отобранной сессии извлеки ключевые темы:
- Заголовки H2/H3 в теле файла.
- Слова из slug имени файла (например, `2026-03-15-claude-auth-refactor` → тема `auth`).
- Ключевые существительные из секции "Что сделано" или "Знания для промоции".

Сгруппируй сессии по пересекающимся темам. Одна сессия может попасть в несколько кластеров.

Используй **concepts** из типизированных наблюдений в таблице Observations (если есть) как дополнительный сигнал кластеризации. Например, несколько сессий с `concepts: ["gotcha"]` по одной теме → кластер «ловушки в [тема]».

**Минимальный кластер**: ≥ 2 сессии с общей темой. Одиночные сессии без пары — занести в раздел «Одиночные сигналы» отчёта, не синтезировать.

## Шаг 3. Добыча повторяющихся сигналов

Для каждого кластера найди факты/наблюдения, которые встречаются независимо в ≥ 2 сессиях кластера.

«Независимо» означает: не copy-paste из одной сессии в другую, а самостоятельное упоминание в разных сессионных файлах.

Присвой каждому сигналу уровень confidence:

| Occurrences | Confidence | Действие                                          |
|-------------|------------|---------------------------------------------------|
|  1 сессия   | `observed` | Не промотировать; занести в одиночные сигналы     |
|  2 сессии   | `inferred` | Промотировать с пометкой ⚠️ (требует верификации) |
|  3+ сессии  | `confirmed` | Промотировать как верифицированный факт          |

## Шаг 4. Синтез инсайтов

Для каждого сигнала с confidence `inferred` или `confirmed` сформулируй **синтезированный инсайт** — не дословное цитирование из сессии, а обобщение:

- Плохо: «В сессии 2026-03-10 было написано: "использовали PostgreSQL"»
- Хорошо: «Проект устойчиво использует PostgreSQL; выбор подтверждён в 3 независимых сессиях»

Каждый инсайт должен содержать:
```
Текст: [обобщённое утверждение]
Тема: [название кластера]
Confidence: observed | inferred | confirmed
Occurrences: N
Sources: [список SESSIONS/файлов]
Предлагаемый владелец: [канонический файл по INDEX.md]
```

## Шаг 5. Проверка перед промоцией

Перед записью каждого инсайта:

0. **Privacy scan:** просканируй инсайт на теги `<private>`, `<sensitive>`, `<ephemeral>`. Применяй `PATTERNS/privacy-control.md`. Контент `<ephemeral>` — не промотировать.
1. **Прочитай файл-владелец** и проверь, нет ли уже этого факта → если есть: `SKIP`.
2. **Проверь на конфликт** с существующим фактом → если конфликт: `CONFLICT`, не промотировать.
3. **Проверь на соответствие CONSTITUTION.md** → если конфликт: `CONSTITUTION_CONFLICT`, не промотировать, только вынести на human review.

## Шаг 6. Промоция по типам инсайтов

Маршрутизируй каждый инсайт в соответствующий канонический файл:

### Инженерный паттерн (повторяемая техника)
→ Создай или обнови `memory-bank/PATTERNS/reflect-<тема>.md`:

```markdown
---
title: "PATTERN — <тема>"
type: reflected
source_sessions:
  - "SESSIONS/<файл1>"
  - "SESSIONS/<файл2>"
confidence: confirmed | inferred
occurrences: N
reflected: ГГГГ-ММ-ДД
last_verified: ГГГГ-ММ-ДД
---

# <Название паттерна>

## Суть
[Синтезированное описание]

## Контекст наблюдения
[Из каких задач получен паттерн]

## Применение
[Когда и как использовать]
```

### Архитектурный факт (как устроена система)
→ Добавь в соответствующую секцию `ARCHITECTURE.md` с провенансом:
```
[факт] <!-- reflected: ГГГГ-ММ-ДД, sources: SESSIONS/... -->
```
Обнови `last_verified`.

### Кандидат в решение (повторяющийся выбор)
→ Добавь строку в `DECISIONS.md` со статусом `💡 Предложено` и примечанием `reflected`:
```
| N | [решение] | 💡 Предложено | ГГГГ-ММ-ДД | — | reflected, N сессий |
```

### Рекурентный открытый вопрос (проблема без решения)
→ Добавь в `OPEN_QUESTIONS.md` с пометкой `reflected: N сессий`.

### Кандидат в принцип конституции (повторяющийся инвариант)
→ **НИКОГДА не записывать в CONSTITUTION.md автоматически.**
→ Занести в отчёт как «Предложения для CONSTITUTION.md» — только для human review.

## Шаг 7. Пометка обработанных сессий

Для каждой сессии, включённой в рефлексию (даже если инсайты из неё не прошли промоцию), добавь в конец файла:

```markdown
---
<!-- reflected: ГГГГ-ММ-ДД -->
```

Это предотвращает повторную обработку в следующих запусках.

## Шаг 8. Отчёт

```
## Рефлексия — [дата]

### Обработано сессий: N
### Кластеров обнаружено: N

### Синтезировано инсайтов: N
- [тема] (confirmed/inferred, N сессий) → [файл-владелец] (ADD/UPDATE/SKIP)

### Конфликты (нужен ревью человека)
- [инсайт] конфликтует с [существующим фактом в файле]

### Предложения для CONSTITUTION.md (только human review)
- [кандидат в принцип]

### Одиночные сигналы (не синтезированы, < 2 сессий)
- [тема]: [кратко] (источник: [файл])

### Помечено сессий как reflected: N
```

## Правила

- Никогда не записывай секреты, токены или PII.
- Рефлексия не заменяет consolidate: факты из отдельных сессий промотирует consolidate.
- Не перезаписывай `CONSTITUTION.md` автоматически — только предлагай.
- Confidence `observed` — только в отчёт, не в канонические файлы.
- Каждый промотированный инсайт содержит список источников (source_sessions).
- При наличии конфликта — фиксируй, но не разрешай самостоятельно.
